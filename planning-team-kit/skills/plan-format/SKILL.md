# plan-format

기획자가 작성한 기획 내용을 받아 먼저 **초안 생성 가능성 검증**을 수행하고, 문서 초안을 만들 수 있는 수준이면 **기능설계서와 정책서 두 문서를 동시에** 생성해 로컬 파일로 저장하는 formatting 중심 스킬.

두 문서 생성은 공통 분석 결과를 기준으로 한다. 입력 해석, 기능명 추출, Draftability gate, 공통 용어·역할·범위 정리는 순차로 확정하고, 템플릿에 들어갈 문서 본문 작성만 기능설계서/정책서로 병렬 실행한다. 저장 전에는 병렬 결과를 단일 흐름에서 다시 조정한다.

`plan-format`은 Confluence 검색 또는 검증 도구가 아니다. 입력값이 문서 생성 가능한 최소 수준인지 사전 점검한 뒤, 통과한 입력만 현재 템플릿 구조에 맞게 정리한다. 세부적으로 부족한 내용은 `[미정]`, `[가정]`, 확인 필요 질문으로 남긴다. **Confluence 검증은 plan-review**에서 수행한다.

입력 형태 무관: AI 대화 결과물, 비구조 기획 노트, 초안 문서 모두 받을 수 있다. 단, Draftability gate를 통과하지 못하면 파일을 저장하지 않고 보완 피드백만 반환한다.

## 호출

- Claude Code: `/planning-team-kit:plan-format <기획 입력 또는 파일경로>`
- Codex: `$plan-format <기획 입력 또는 파일경로>`

결과물 입력 방식:
- 인자에 직접 붙여넣기
- 파일 경로 전달 (예: `/path/to/output.md`)

기능명은 별도 인자로 받지 않는다. 입력 본문 또는 파일명에서 기능명을 추출한다.

## 동작 순서

1. 입력 받기
2. 기능명 추출
3. 초안 생성 가능성 검증
4. 부족하면 저장하지 않고 피드백 반환
5. 충분하면 입력 내용을 기능설계서/정책서로 분류
6. 템플릿에 맞춰 채우기
7. 일부 미확정은 `[미정]` / `[가정]` / 확인 필요 질문으로 남기기
8. 파일 저장

병렬화 원칙:
- 공통 분석은 순차 실행한다.
- 문서 본문 작성만 병렬 실행한다.
- 최종 조정은 단일 실행한다.

### 1. 입력 받기

- 인자가 파일 경로이면 파일 읽기
- 아니면 인자 또는 대화 컨텍스트에서 붙여넣은 내용 사용
- 입력이 없으면 붙여넣기 요청 후 대기

입력 형태 제약 없음: 비구조 메모, 회의록 수준의 초안, AI 대화 결과물, 기존 문서 스크랩 모두 허용. 단, 이후 Draftability gate에서 문서 생성 가능 여부를 판정한다.

### 2. 기능명 추출

입력 내용에서 기능명을 추출한다. 사용자에게 기능명을 별도로 묻지 않는다.

추출 우선순위:
1. 문서 제목 또는 명시된 기능명
2. 반복 등장하는 `~ 기능`, `~ 정책`, `~ 화면`, 메뉴명
3. 파일 입력인 경우 파일명
4. 첫 번째 핵심 명사구

그래도 확정하기 어려우면 `기획초안`을 기능명으로 사용한다.

### 3. 초안 생성 가능성 검증

Draftability gate는 생성 전 판단이다. 입력이 기능설계서/정책서 초안으로 구조화될 최소 정보를 갖췄는지 확인한다.

저장 가능 조건:
- 기능 목적/기능명 식별 가능
- 적용 대상 또는 업무 범위 식별 가능
- 핵심 사용자 행동과 기대 결과 식별 가능
- 주요 조건/정책/제약 식별 가능

저장 보류 조건:
- 저장 가능 조건 중 하나라도 문서화할 수 없을 정도로 없음
- 기능설계서/정책서로 나눌 핵심 내용이 거의 없음
- 입력이 단순 키워드, 제목, 요청 의도만 있고 기능 동작이나 정책 판단 근거가 없음

예외/권한/상태/근거/알림 등 세부 항목은 부족해도 저장 가능하다. 이 경우 문서 안에 `[미정]`, `[가정]`, 확인 필요 질문으로 남긴다.

### 4. 부족하면 저장하지 않고 피드백 반환

Draftability gate가 `blocked`이면 파일을 만들지 않는다. 부족 항목과 입력 보완 질문만 반환한다.

출력 형식:

```text
저장 보류
- 추출 기능명: [추출기능명]
- Draftability gate: blocked
- 이유: 입력만으로 기능설계서/정책서 초안을 만들기 부족함

부족 항목:
- [항목]: [부족한 이유]

입력 보완 질문:
1. [질문]
2. [질문]
3. [질문]
```

### 5. 충분하면 입력 내용을 기능설계서/정책서로 분류

입력 전체를 스캔해 두 문서에 해당하는 내용을 분리한다.

| 분류 기준 | 귀속 문서 |
|:---------|:---------|
| 화면·기능·플로우·HOW·입력·UI·기능 동작 명세 | 기능설계서 |
| 규칙·조건·예외·정책·기준·제한·원칙 | 정책서 |

같은 문장이 양쪽에 해당할 때: 비즈니스 규칙 측면 → 정책서, 기능 동작 측면 → 기능설계서로 나눠 배치.

Draftability gate 통과 시 두 문서를 모두 생성한다. 한 문서만 생성하지 않는다.

### 6. 템플릿에 맞춰 채우기

먼저 공통 planning brief를 만든다. 공통 planning brief에는 두 문서가 공유해야 하는 기능명, 적용 범위, 핵심 사용자 행동, 주요 정책·제약, 공통 용어, 역할명, `[미정]`, `[가정]`, 확인 필요 질문 후보를 포함한다.

공통 planning brief가 만들어진 뒤에만 문서 본문 작성만 병렬 실행한다.

- 기능설계서 작성 worker: 기능설계서 템플릿(`skills/plan-format/templates/기능설계서.md`)을 기준으로 화면·기능·플로우·HOW·입력·UI·기능 동작 명세를 작성한다.
- 정책서 작성 worker: 정책서 템플릿(`skills/plan-format/templates/정책서.md`)을 기준으로 규칙·조건·예외·정책·기준·제한·원칙을 작성한다.

각 worker는 공통 planning brief와 원 입력만 근거로 사용한다. worker끼리 기능명, 역할명, 범위, 확정/미정 상태를 새로 판단하지 않는다.

해당 문서 타입 템플릿(`skills/plan-format/templates/`) 기반으로 입력 내용을 섹션별로 매핑한다.

입력이 비구조적이어도 전체 입력을 한 번 스캔한 뒤 관련 내용을 각 섹션으로 분산 매핑한다. 같은 내용이 여러 섹션에 해당하면 가장 적합한 섹션 하나에만 배치한다.

- 입력에서 확인 가능한 내용 → 섹션 채우기
- 매핑 안 된 필수 섹션 → `[미정]`
- 입력에 없지만 컨텍스트로 추론 가능 → `[가정]` 태그 명시
- 기능설계서 §5, 정책서 §9·§10의 역할명은 입력에서 감지된 도메인에 맞게 **두 문서 동시에, 동일한 역할명으로** 치환한다. 두 문서의 역할명이 달라지면 안 된다. 감지 불가 시 기본값(물류/WMS: 입고담당자·창고관리자·운송팀·화주) 유지.

병렬 작성 결과가 나오면 최종 조정은 단일 실행한다. 저장 전 다음 항목을 확인하고 필요하면 조정한다.

- 역할명 불일치 여부
- 기능명·적용 범위·공통 용어 불일치 여부
- 같은 내용이 두 문서에 부적절하게 중복되었는지 여부
- `[미정]`, `[가정]`, 확인 필요 질문이 출력 형식에 맞게 문서별로 취합되었는지 여부

### 7. 일부 미확정은 `[미정]` / `[가정]` / 확인 필요 질문으로 남기기

질문 루프 없음. 모든 미확인 항목은 `[미정]` / `[가정]` / 확인 필요 질문으로 저장한다.

### 8. 파일 저장

두 문서를 동일 폴더에 저장한다.

저장 경로:
- `planning/[추출기능명]/[추출기능명]_기능설계서.md`
- `planning/[추출기능명]/[추출기능명]_정책서.md`

**기존 파일 있을 때 (파일별 확인):**
```
planning/[추출기능명]/[추출기능명]_기능설계서.md 이 이미 존재합니다.
planning/[추출기능명]/[추출기능명]_정책서.md 이 이미 존재합니다.

1. 둘 다 덮어쓰기
2. 기능설계서만 덮어쓰기
3. 정책서만 덮어쓰기
4. 취소

선택하세요.
```

폴더가 없으면 생성 후 저장.

## 출력 형식

Draftability gate가 `passed`이면 저장 완료 형식으로 출력한다.

```
저장 완료
- 추출 기능명: [추출기능명]
- Draftability gate: passed
- planning/[추출기능명]/[추출기능명]_기능설계서.md
- planning/[추출기능명]/[추출기능명]_정책서.md

[확인 필요 질문] (N개):
- [질문]: [결정 필요 이유]

[기능설계서] [미정] 항목 (N개):
- [섹션명]: 확인 필요

[기능설계서] [가정] 항목 (N개):
- [섹션명]: [가정 내용]

[정책서] [미정] 항목 (N개):
- [섹션명]: 확인 필요

[정책서] [가정] 항목 (N개):
- [섹션명]: [가정 내용]

다음 단계: /planning-team-kit:plan-review planning/[추출기능명]/[추출기능명]_기능설계서.md
```

## plan-review 권장 신호

Draftability gate는 생성 전 판단이다. 발행 가능성 판단은 `plan-review` 책임이다.

다음 항목이 있으면 출력에 남기고, 발행 전 `plan-review` 실행을 안내한다.
- `[미정]`
- `[가정]`
- 확인 필요 질문

## 규칙

- Draftability gate 통과 시 두 문서를 모두 생성한다. 한 문서만 생성하지 않는다
- Draftability gate가 blocked이면 저장 보류 피드백만 반환하고 파일을 만들지 않는다
- 상위설계서는 지원하지 않는다
- 기능명은 입력받지 않는다. 입력 내용에서 기능명을 추출한다
- 질문 루프 없이 단일 패스로 변환한다
- 공통 분석은 순차 실행하고, 문서 본문 작성만 병렬 실행하며, 최종 조정은 단일 실행한다
- Confluence 문서 검색이나 index 조회를 수행하지 않는다
- Confluence 검증은 plan-review 책임이다
- 기존 Confluence 내용과의 충돌 여부를 plan-format에서 확정하지 않는다
- 개발자 소유 API 계약·내부 구현 결정은 기능설계서에 생성하지 않는다
- 초안은 항상 로컬에만 저장된다. Confluence 반영은 이 스킬 범위 밖의 별도 절차에서 한다
- 저장 경로는 `planning/[추출기능명]/[추출기능명]_기능설계서.md`, `planning/[추출기능명]/[추출기능명]_정책서.md` 형식을 벗어나지 않는다

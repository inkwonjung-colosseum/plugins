# product-team-kit Examples

## 기능설계서/정책서 동시 생성

```text
$plan-format "주문 취소 기능: 주문 취소 가능 조건, 화면 흐름, 예외 처리 메모..."
```

기능명을 입력하지 않는다. 스킬이 입력 본문에서 `주문 취소`를 추출한다.

예상 흐름:

1. 입력 본문에서 기능명과 핵심 요구를 추출한다.
2. Draftability gate로 문서 생성 가능한 최소 입력인지 확인한다.
3. 충분하면 공통 planning brief로 기능명, 범위, 역할명, 공통 용어를 확정한다.
4. 기능설계서와 정책서 본문만 병렬로 작성한다.
5. 단일 흐름에서 역할명·범위·미정 항목을 최종 조정한다.
6. 부족한 세부 조건은 질문하지 않고 `[미정]`, `[가정]`, 확인 필요 질문으로 남긴다.
7. 추출한 기능명 기준으로 `planning/주문취소/주문취소_기능설계서.md`, `planning/주문취소/주문취소_정책서.md`를 저장한다.

입력이 부족하면 파일을 만들지 않고 아래처럼 저장 보류 피드백을 반환한다.

```text
저장 보류
- Draftability gate: blocked
- 이유: 입력만으로 기능설계서/정책서 초안을 만들기 부족함

부족 항목:
- 적용 대상: 어떤 사용자나 업무 범위에 적용되는지 없음
- 핵심 동작: 사용자가 수행하는 행동과 기대 결과가 없음

입력 보완 질문:
1. 이 기능은 누가 사용하나요?
2. 사용자는 어떤 행동을 수행하나요?
3. 성공 또는 실패 시 어떤 결과가 나와야 하나요?
```

문서가 저장되면 출력에 `Draftability gate: passed`가 포함된다. 초안에 `[미정]`, `[가정]`, 확인 필요 질문이 있으면 발행 전 `plan-review`로 검토한다.

## 초안 검토

```text
$plan-review planning/주문취소/주문취소_기능설계서.md
```

`plan-review`는 4개의 병렬 fresh-context reviewer가 같은 파일을 직접 읽고 관점별 verdict를 낸 뒤, 가장 보수적인 verdict를 최종 결과로 사용한다.

검토 관점:

- 근거: 기존 Confluence 문서와 충돌하는지 확인한다.
- 범위: 결정해야 할 정책과 예외가 빠졌는지 확인한다.
- 실행: 상태, 조건, 예외, 사용자 노출 결과가 명확한지 확인한다.
- 템플릿 준수: 정책 ID와 인수 조건 / 확인 기준이 필요한 수준으로 채워졌는지 확인한다.

결과는 `pass`, `conditional pass`, `수정 필요` 중 하나다.

## 파일 입력

```text
$plan-format /path/to/입고신청-planning-notes.md
```

파일 경로를 넘기면 해당 파일 내용을 읽고, 파일명과 본문에서 기능명을 추출해 같은 방식으로 기능설계서와 정책서 초안을 만든다.

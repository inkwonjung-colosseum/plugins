# product-team-kit Quality Rubric

`plan-review`는 `plan-format`이 저장한 기능설계서와 정책서를 발행 전에 근거, 결정·범위, 실행·검증 가능성 관점에서 확인한다. 각 관점은 현재 대화 컨텍스트가 아니라 검토 대상 파일과 필요한 근거 문서를 직접 읽어 판단한다.

`plan-review`는 템플릿 구조 검증 도구가 아니다. 발행 전에 근거가 맞는지, 결정 범위가 충분한지, 개발·운영·QA가 같은 기준으로 움직일 수 있는지만 확인한다.

`plan-format`은 formatter다. 먼저 초안 생성 가능성을 점검하고, 통과한 입력만 템플릿에 맞게 구조화한다. Confluence 충돌 검증과 발행 가능성 판단은 `plan-review`에서 수행한다.

입력이 기능설계서/정책서 초안을 만들 최소 수준에 못 미치면 `plan-format`은 저장 보류를 반환하고 파일을 만들지 않는다.

## 검토 결과

| 결과 | 의미 | 발행 판단 |
|---|---|---|
| `pass` | 치명적인 누락이나 충돌이 없다 | 발행 후보 |
| `conditional pass` | 남은 조건을 기획자가 인지하고 확인하면 반영 가능하다 | 조건 확인 후 발행 후보 |
| `수정 필요` | 구현 또는 운영 판단에 필요한 정보가 부족하다 | 불가 |

최종 결과는 관점별 결과 중 가장 보수적인 값을 사용한다: `수정 필요 > conditional pass > pass`.

## 발견 사항 분류

`plan-review`는 구조화된 발견 사항을 사용해 검토 결과를 합성한다. 발견 사항은 `관점`, `위치`, `발견 유형`, `근거 인용`, `영향`, `최소 수정 포인트 또는 확인 조건`, `출력 버킷`을 가진다.

- 발견 유형은 `오류`와 `누락`으로 구분한다.
- 신뢰도 앵커: 50 / 75 / 100.
- 출력 버킷: 수정 포인트 / 확인 조건 / 참고 관찰.
- 수정 포인트: 발행 전 문서 수정 없이는 구현 또는 운영 판단이 달라지는 항목.
- 확인 조건: 기획자가 명시적으로 확인하거나 수용하면 발행 후보가 될 수 있는 항목.
- 참고 관찰: 사실 기반 관찰이지만 발행 판단을 낮추지 않는 항목.
- 참고 관찰은 최종 검토 결과를 낮추지 않는다.

## 근거

확인 항목:

- 기존 Confluence 문서와 충돌하지 않는다.
- 출처가 필요한 주장은 관련 문서나 회의록을 근거로 가진다.
- `[가정]` 항목은 실제 결정이 필요한지 구분되어 있다.
- archive 문서는 현재 정책처럼 사용하지 않는다.
- 초안의 근거는 입력값에 명시된 내용과 별도 검토 문서로 구분되어 있다.
- 직접 관련 문서뿐 아니라 상위/하위 문서, 참조 문서, sibling 문서 중 충돌 가능성이 있는 관련 근거를 충분히 확장해 읽는다.
- 읽지 않은 관련 후보와 검증 한계를 남긴다.

## 결정·범위

확인 항목:

- 문서 타입이 기능설계서, 정책서 중 하나로 명확하다.
- 적용 대상, 비대상, 예외가 분리되어 있다.
- 확정된 결정의 적용/비적용/예외 범위가 닫혀 있다.
- 관련 정책서/기능설계서가 필요한 항목은 문서명이나 경로로 연결된다.
- `[미정]` 항목은 검토 결과를 낮추는 근거로 사용하지 않는다.
- 부서 소유 상세 산출물이 본문에 완성본처럼 섞였는지 확인하고, 필요하면 관련 문서 또는 확인 필요 사항으로 분리한다.

## 실행·검증 가능성

확인 항목:

- 상태, 조건, 권한, 사용자 노출 결과, 예외 처리 기준이 명확하다.
- 정상 기준과 주요 예외 기준이 확인 기준으로 있다.
- 수정 문서라면 `[기존]`과 `[변경]` 차이가 명확하다.
- 후속 검증에서 확인할 수 있는 체크포인트가 있다.
- 업무 연동 경계, 업무 데이터, 외부 채널, 실패 대응, 운영 영향이 필요한 수준으로 드러난다.

## Review Gate

발행 가능성 판단은 `plan-review`에서 수행한다. 아래 항목 중 하나라도 남아 있으면 발행 전에 검토 결과 또는 명시적 의사결정이 필요하다.

- `[가정]`
- 확인 필요 질문
- 기존 Confluence 문서와의 충돌 경고

## 초안 생성 가능성

초안 생성 가능성 검증은 생성 전 판단이다.

- `초안 생성 가능: 예`: 기능 목적/기능명, 적용 대상 또는 업무 범위, 핵심 사용자 행동과 기대 결과, 주요 조건/정책/제약이 식별되어 문서 생성 가능.
- `초안 생성 가능: 아니오`: 문서 초안을 만들 최소 입력이 부족하여 저장 보류.
- 예외, 권한, 상태, 근거, 알림 같은 세부 항목은 부족해도 문서 생성 가능하며, 생성된 문서 안에 `[미정]`, `[가정]`, 확인 필요 질문으로 남긴다.

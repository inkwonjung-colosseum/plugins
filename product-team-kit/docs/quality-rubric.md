# product-team-kit Quality Rubric

`plan-review`는 `plan-format`이 저장한 기능설계서 또는 정책서 파일을 발행 전에 근거, 범위, 실행 가능성 관점에서 확인한다. 각 관점은 병렬 fresh-context reviewer가 담당하며, 현재 대화 컨텍스트가 아니라 검토 대상 파일과 필요한 근거 문서를 직접 읽어 판단한다.

`plan-format`은 formatter다. 먼저 `Draftability gate`로 초안 생성 가능성을 점검하고, 통과한 입력만 템플릿에 맞게 구조화한다. Confluence 충돌 검증과 발행 가능성 판단은 `plan-review`에서 수행한다.

입력이 기능설계서/정책서 초안을 만들 최소 수준에 못 미치면 `plan-format`은 저장 보류를 반환하고 파일을 만들지 않는다.

## Verdicts

| Verdict | 의미 | 발행 판단 |
|---|---|---|
| `pass` | 치명적인 누락이나 충돌이 없다 | 발행 후보 |
| `conditional pass` | 남은 조건을 기획자가 인지하고 확인하면 반영 가능하다 | 조건 확인 후 발행 후보 |
| `수정 필요` | 구현 또는 운영 판단에 필요한 정보가 부족하다 | 불가 |

최종 verdict는 관점별 verdict 중 가장 보수적인 값을 사용한다: `수정 필요 > conditional pass > pass`.

## 근거

확인 항목:

- 기존 Confluence 문서와 충돌하지 않는다.
- 출처가 필요한 주장은 관련 문서나 회의록을 근거로 가진다.
- `[가정]` 항목은 실제 결정이 필요한지 구분되어 있다.
- archive 문서는 현재 정책처럼 사용하지 않는다.
- 초안의 근거는 입력값에 명시된 내용과 별도 검토 문서로 구분되어 있다.

## 범위

확인 항목:

- 문서 타입이 기능설계서, 정책서 중 하나로 명확하다.
- 적용 대상, 비대상, 예외가 분리되어 있다.
- `[미정]` 항목은 결정 필요자나 후속 액션이 있다.
- 관련 정책 또는 연관 기능설계서/정책서와 연결된다.

## 실행

확인 항목:

- 상태, 조건, 권한, 사용자 노출 결과, 예외 처리 기준이 명확하다.
- 정상 기준과 주요 예외 기준이 인수 조건 / 확인 기준으로 있다.
- 수정 문서라면 `[기존]`과 `[변경]` 차이가 명확하다.
- 후속 검증에서 확인할 수 있는 체크포인트가 있다.

## Review Gate

발행 가능성 판단은 `plan-review`에서 수행한다. 아래 항목 중 하나라도 남아 있으면 발행 전에 review 결과 또는 명시적 의사결정이 필요하다.

- `[미정]`
- `[가정]`
- 확인 필요 질문
- 기존 Confluence 문서와의 충돌 경고

## Draftability Gate

`Draftability gate`는 생성 전 판단이다.

- `Draftability gate: passed`: 기능 목적/기능명, 적용 대상 또는 업무 범위, 핵심 사용자 행동과 기대 결과, 주요 조건/정책/제약이 식별되어 문서 생성 가능.
- `Draftability gate: blocked`: 문서 초안을 만들 최소 입력이 부족하여 저장 보류.
- 예외, 권한, 상태, 근거, 알림 같은 세부 항목은 부족해도 문서 생성 가능하며, 생성된 문서 안에 `[미정]`, `[가정]`, 확인 필요 질문으로 남긴다.

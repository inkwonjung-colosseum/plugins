# product-team-kit Style Guide

## 문서 원칙

- Confluence 원본을 source of truth로 둔다.
- 로컬 Markdown export는 읽기 전용 snapshot으로 취급한다.
- 확인된 사실, 가정, 미정 항목을 섞지 않는다.
- `plan-format`은 생성 전 초안 생성 가능성을 점검하고, 통과한 입력만 템플릿에 맞게 정리한다.
- `plan-format`은 입력 해석과 공통 정리 기준 확정은 순차로 처리하고, 기능설계서/정책서 본문 작성만 병렬화한다.
- `plan-format`은 Confluence 검색이나 index 조회를 수행하지 않는다.
- Confluence 근거 검증과 발행 전 결정·실행 가능성 판단은 `plan-review`에서 수행한다.
- 기획자가 결정해야 하는 기능 동작, 조건, 예외를 읽기 좋은 형식으로 정리한다.
- Confluence 발행 문서는 정책/기능 판단 계약을 소유한다.
- 부서별 상세 산출물은 본문에 완성본처럼 생성하지 않는다.
- 부서별 상세 산출물이 필요하면 관련 문서 또는 확인 필요 사항으로만 남긴다.
- 표는 조건, 권한, 상태, 예외, 확인 기준처럼 비교가 필요한 곳에만 사용한다.
- Mermaid는 표시 가능한 흐름, 상태, 분기, 관계 구조가 있으면 기본적으로 추가한다.
- Mermaid는 표나 본문에 이미 있는 흐름·상태·분기·관계만 요약하며 새 사실을 만들지 않는다.
- Mermaid가 표나 본문과 불일치하면 제거하거나 수정한다.

## 표현

- 확인되지 않은 내용은 `[가정]`으로 시작한다.
- 결정되지 않은 내용은 `[미정]`으로 표시한다.
- 검증이 필요한 내용은 확인 필요 질문으로 남긴다.
- 기존 문서와의 충돌 판정은 `plan-review`에서 수행한다.
- 정책 조건은 "조건 → 처리 방식 → 비고" 순서로 쓴다.

## 문서 타입

| 타입 | 사용 기준 |
|---|---|
| 기능설계서 | 개요, 범위, 진입점, 사용자 흐름, 화면·입력 항목, 기능 동작, 권한, 예외, 확인 기준을 정리한다 |
| 정책서 | 정책 목적, 적용 범위, 비즈니스 규칙, 조건, 상태·처리 기준, 역할·권한, 예외·승인 기준을 정리한다 |

## 초안 출력 문구

초안 출력에는 생성 전 판단인 초안 생성 가능성만 포함한다. 발행 전 근거·결정·실행 가능성 판단은 `plan-review`에서 수행한다.

입력이 부족하면 파일을 만들지 않고 저장 보류를 출력한다.

```text
저장 보류
초안 생성 가능: 아니오
```

입력이 충분해 파일을 만들면 다음처럼 출력한다.

```text
초안 생성 가능: 예
```

또는:

```text
초안 생성 가능: 예
plan-review 권장 신호: [가정], 확인 필요 질문
```
